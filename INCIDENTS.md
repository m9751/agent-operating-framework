# Incident Log

Every rule in this framework was born from a real failure. This log is the source of record.

Each row shows the incident, the lesson, and the rule it hardened. Dates are month-precision. Names of systems, accounts, and employers have been genericized — the failures are the point, not the forensics.

| # | Month | What happened | Lesson | Rule |
|---|-------|--------------|--------|------|
| 1 | 2026-03 | Edited a config file that wasn't tracked in the project. Full edit cycle wasted finding the right file afterward. | Read the file's current state before touching it — every session, not from memory. | [read-before-acting](examples/claude-code-rules/read-before-acting.md) Gate 0 |
| 2 | 2026-03 | Entire session spent chasing the wrong hypothesis about a crash. Reading 43 lines of config found the real problem in 2 minutes. | Acting without reading wastes hours. Research is the work; the fix is the easy part. | [read-before-acting](examples/claude-code-rules/read-before-acting.md) Gate 0 |
| 3 | 2026-03 | Guessed an API's required format, property names, and deploy model. Docs were found in under 2 minutes. Over an hour of session time burned. | If a tool exists to check it — `--help`, `curl`, schema query — use it. Don't ask the user. | [read-before-acting](examples/claude-code-rules/read-before-acting.md) Gate 3 |
| 4 | 2026-03 | Wrong deployment system chosen. A JAR-based asset deployed against a completely different runtime. Never flagged "I've never done this type of deployment before." | Identify and name first-time actions before acting on them. Say it out loud. | [read-before-acting](examples/claude-code-rules/read-before-acting.md) Gate 1 |
| 5 | 2026-03 | 5+ failed database queries on columns that don't exist. The schema was readable in the database the entire time. | Never guess column names. Schema is always queryable. | [read-before-acting](examples/claude-code-rules/read-before-acting.md) Gate 3 |
| 6 | 2026-03 | CLI install commands recommended from a third-party README. Syntax was wrong. When challenged, agreed they were fake — without checking either claim. Both the confidence and the retraction were unverified. | Verify CLI commands with `--help` before recommending. Never retract without checking. | [read-before-acting](examples/claude-code-rules/read-before-acting.md) Gate 3 |
| 7 | 2026-03 | Same deployment task failed 5+ times. The right answer — a different runtime — was in the documentation and found in 5 minutes after stopping. An hour of late-night session burned. | Stop after 3 failures. State what was tried. Research before the next attempt. | [read-before-acting](examples/claude-code-rules/read-before-acting.md) Three-Failure Stop |
| 8 | 2026-03 | An API secret was shared in plain text in the chat window during a session. | Never accept API keys or tokens in chat. Use a credential manager. | [secure-configuration](examples/claude-code-rules/secure-configuration.md) |
| 9 | 2026-03 | A shared config file overwritten with a full-replace write tool instead of an edit. Done four times in one session before the session was terminated. | On shared config files: Edit only, never Write. Take a backup first. | [secure-configuration](examples/claude-code-rules/secure-configuration.md) |
| 10 | 2026-03 | Automation scripts shipped without a required menu hook. Happened twice before the invariant was formalized. | Ship-time checklists must run before "done" is said. Not after. | [delivery-protocol](examples/claude-code-rules/delivery-protocol.md) |
| 11 | 2026-03 | Six integration resources built from scratch. All six had production-ready equivalents in a shared asset catalog that was never searched. | Search before building. Reuse what exists. The catalog search is the first step, not an afterthought. | [scope-discipline](examples/claude-code-rules/scope-discipline.md) Gate 3 |
| 12 | 2026-03 | A platform integration demo was delivered with zero instances of the target platform running. All logic ran in a different language on a different runtime. | When demonstrating a platform's capabilities, use that platform. | [scope-discipline](examples/claude-code-rules/scope-discipline.md) Gate 4 |
| 13 | 2026-03 | Required auth parameters stripped from a working OAuth flow based on a handoff note asserting they were unnecessary — without verifying. Both were required. 3 extra deploy cycles. Flow broken for 5 hours. | Never remove parameters from a working API call without running the full flow without them first. Handoff notes are hypotheses, not facts. | [read-before-acting](examples/claude-code-rules/read-before-acting.md) Gate 3a |
| 14 | 2026-03 | Wrong credentials deployed to wrong system. Config file was not labeled with its target app. | Label every external config file with its target. Verify match before deploying. | [secure-configuration](examples/claude-code-rules/secure-configuration.md) |
| 15 | 2026-04 | 5 persistent agents built on a local machine — no backup, no redundancy, no operation without the machine being on. A cloud-hosted alternative with 247k GitHub stars existed and was never searched. | Before building any persistent infrastructure, ask: "does this still run if this machine is off for a week?" If no, find a cloud alternative first. | [no-local-infrastructure](examples/claude-code-rules/no-local-infrastructure.md) |
| 16 | 2026-04 | Deliverable files scattered across 4 different storage systems across multiple sessions. Every session invented a different delivery path. | Define one delivery workflow and follow it every time. One path, not four. | [delivery-protocol](examples/claude-code-rules/delivery-protocol.md) |
| 17 | 2026-04 | A post-delivery checklist was defined but skipped 10+ times across sessions. | Checklists must run before "done" is said. Defining a checklist is not the same as running it. | [delivery-protocol](examples/claude-code-rules/delivery-protocol.md) |
| 18 | 2026-04 | Infrastructure change planned in 8 careful phases. The deployment step was improvised. Three failed push attempts. The user had to run the final command manually. | Deployment requires the same execution-protocol rigor as the build. Write the deployment sub-plan before the build starts. | [delivery-protocol](examples/claude-code-rules/delivery-protocol.md) |
| 19 | 2026-04 | 8-phase remediation plan proposed for a file with zero callers outside itself. Framed as urgent. The code was dormant — the work would have been invisible to the system. | Before proposing any remediation plan, grep for callers. Zero callers = dormant code. Ceremony must match blast radius. | [scope-discipline](examples/claude-code-rules/scope-discipline.md) Gate 5 |
| 20 | 2026-04 | Analysis script simulated a constraint the actual system does not have. Simulation output presented as a real finding. The user built plans on a fabricated bottleneck. | Scripts that simulate system behavior must be verified against the actual system code. Numbers from a script are only as accurate as the assumptions baked into it. | [read-before-acting](examples/claude-code-rules/read-before-acting.md) Gate 4 |
| 21 | 2026-04 | A directory's full contents claimed from a filtered file search. The claim missed two entire sub-repos. Filtered search is not a listing. | When asserting what a directory contains, run an unfiltered listing. Filtered searches prove what matches, not what exists. | [read-before-acting](examples/claude-code-rules/read-before-acting.md) Gate 4 |
| 22 | 2026-04 | Verification done via file-existence checks and grep. Called it "confirmed wiring." Zero reads of actual artifact content. | Proxy checkpoints prove existence, not function. Verification requires reading the artifact itself or actively invoking the behavior. | [read-before-acting](examples/claude-code-rules/read-before-acting.md) Gate 4 |
| 23 | 2026-04 | Persistent memory grew past the agent's auto-load window. Critical context was invisible every session for weeks without anyone noticing. | Set a size gate on persistent memory files. Anything past the auto-load limit is invisible context. | [session-lifecycle](examples/claude-code-rules/session-lifecycle.md) |
